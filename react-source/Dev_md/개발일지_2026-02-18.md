# 개발일지 — 2026-02-18

**프로젝트**: DreamIT Biz React SPA (www.dreamitbiz.com)
**개발 단계**: 3차 개발 — 이커머스 (장바구니 + 결제)
**개발자**: Claude Opus 4.6
**소요 시간**: 1세션 (Phase 1~6 전체 완료)

---

## 1. 작업 개요

DreamIT Biz React SPA에 이커머스 기능을 추가했다.
기존 출판(책/전자출판/간행물) 상세 페이지와 교육(강좌) 페이지에는 상품 정보만 있고 가격이나 구매 기능이 없었다.
이번 3차 개발로 **상품 가격 표시 → 장바구니 담기 → 결제(KG이니시스) → 주문 완료**까지의 전체 구매 플로우를 구현했다.

## 2. 기술 결정 사항

| 항목 | 선택 | 사유 |
|------|------|------|
| 결제 PG | PortOne V2 → KG이니시스 | 사용자가 이미 가맹점 보유 |
| DB/서버 | Supabase (PostgreSQL + Edge Functions) | Node.js 서버 불필요, 서버리스 구조 |
| 상태관리 | React Context + localStorage | 별도 라이브러리 없이 경량 구현 |
| 결제검증 | Supabase Edge Function | 클라이언트에서 금액 조작 방지 |
| 데모 모드 | env 미설정 시 localStorage fallback | Supabase/PortOne 없이도 프론트엔드 테스트 가능 |

## 3. 구현 상세

### Phase 1: 데이터 레이어 (상품 데이터 + 장바구니 상태)

**데이터 파일 확장**

기존 `publishingDetails.js`의 `sampleBooks` 배열에는 `title`, `author`, `category`만 있었다.
`products` 배열로 변경하고 `id`, `price`, `isbn` 필드를 추가했다.
같은 방식으로 `ebook.products`(4개), `periodical.products`(4개)도 신규 추가.

`educationDetails.js`의 `sampleCourses` → `courses`로 이름 변경 + `id`, `price` 추가.
이름 변경에 따라 `Classroom.jsx`에서 `data.sampleCourses` → `data.courses`로 참조도 수정.

임시 가격:
- 도서: 28,000~38,000원
- 전자출판: 15,000~22,000원
- 간행물: 18,000~30,000원
- 강좌: 380,000~550,000원

**CartContext 설계**

`ThemeContext` 패턴을 따라 `CartContext.jsx`를 생성했다.

```
Provider: CartProvider
Hook: useCart()
상태: cartItems (배열), cartTotal (합계금액), cartCount (총수량)
액션: addItem, removeItem, updateQuantity, clearCart
영속: localStorage 키 'dreamitbiz_cart'
```

`addItem` 호출 시 이미 장바구니에 있는 상품이면 `quantity + 1`, 없으면 새로 추가.
`App.jsx`에서 `ThemeProvider > LanguageProvider > CartProvider > Router` 순으로 래핑.

**번역 키 추가**

`translations.js`에 `shop`, `cart`, `checkout`, `order` 네 개 네임스페이스를 한/영 모두 추가.
총 약 50개 키. 기존 `common`, `nav` 등과 동일한 구조.

**Shop.jsx 상품 목록 구성**

4개 데이터 소스(book.products, ebook.products, periodical.products, classroom.courses)를 합쳐서 `allProducts` 배열을 만들고 각 항목에 `category` 필드를 부여.

카테고리별 SVG 아이콘 함수 `getCategoryIcon()` 구현.
"장바구니 담기" 클릭 시 1.5초간 "담기 완료" 텍스트로 변경되는 피드백.

**Checkout.jsx 결제 플로우**

```
1. 폼 입력 (이름, 이메일, 전화번호)
2. 결제수단 선택 (카드 / 계좌이체)
3. 동의 체크박스
4. "결제하기" 클릭
5. Supabase에 orders + order_items INSERT (status: pending)
6. PortOne.requestPayment() 호출
7. 성공 시 → clearCart() → /order-confirmation?orderNumber=xxx
8. 실패 시 → 에러 메시지 표시
```

주문번호 형식: `DIT-YYMMDD-XXXXXX` (예: DIT-260218-A3B7CF)

### Phase 4: Supabase 백엔드

**supabase.js 클라이언트**

`VITE_SUPABASE_URL`과 `VITE_SUPABASE_ANON_KEY`가 설정되지 않으면 Supabase 클라이언트를 생성하지 않고 **localStorage fallback** 모드로 동작.
이를 통해 Supabase 프로젝트 없이도 프론트엔드 전체 플로우를 테스트할 수 있다.

주요 함수:
- `createOrder(orderData)` — orders + order_items INSERT
- `getOrderByNumber(orderNumber)` — 주문 조회 (items JOIN)
- `updateOrderStatus(orderId, status, paymentId)` — 상태 변경
- `verifyPayment(paymentId, orderId)` — Edge Function 호출

**DB 스키마 (supabase_schema.sql)**

3개 테이블:
- `products`: id(UUID), title, title_en, category(CHECK), price(INTEGER, 원), is_active
- `orders`: id(UUID), order_number(UNIQUE), user_email, total_amount, payment_status(CHECK), portone_payment_id
- `order_items`: id(UUID), order_id(FK), product_title, quantity, unit_price, subtotal

RLS 정책:
- products: 누구나 SELECT (is_active = true)
- orders/order_items: 누구나 INSERT (비인증 구매 지원)

14개 샘플 데이터 포함 (도서 4, 전자출판 2, 간행물 2, 강좌 6).

**Edge Functions (Deno)**

`verify-payment`: PortOne REST API(`GET /payments/{paymentId}`)로 실제 결제 금액을 조회하고 DB 주문 금액과 비교. 일치 시 `paid`, 불일치 시 `failed` 처리.

`portone-webhook`: PortOne에서 보내는 웹훅(`Transaction.Paid`, `Transaction.Cancelled`, `Transaction.Failed`)을 수신하여 DB 상태 동기화. 웹훅 시그니처 검증 로직 포함(선택적).

## 5. 설계 판단 및 트레이드오프

**1) localStorage fallback을 넣은 이유**

Supabase 프로젝트 설정 전에도 프론트엔드 전체 흐름을 테스트할 수 있어야 한다. `supabase.js`의 `getSupabase()`가 null을 반환하면 `createOrder`, `getOrderByNumber` 등이 자동으로 localStorage 기반으로 동작한다. 배포 시 `.env`만 채우면 Supabase로 전환된다.

**2) 상품 데이터를 DB가 아닌 정적 파일에 둔 이유**

현재 상품 수가 18개로 적고, 관리자 대시보드가 없는 상태. 정적 데이터 파일에서 관리하는 것이 간단하다. 추후 상품이 많아지면 Supabase `products` 테이블에서 fetch하도록 전환 가능. DB 스키마는 이미 준비되어 있다.

**3) 인증 없이 주문 허용**

현재 Supabase Auth 회원 시스템이 구현되지 않은 상태이므로, 주문 시 이메일/이름/전화번호만 입력받는다. RLS 정책도 INSERT는 누구나 가능하도록 열어둠. 회원 시스템 추가 시 `orders.user_id` 컬럼과 RLS 정책을 업데이트하면 된다.

**4) PortOne V2를 동적 import하는 이유**

SDK가 설치되지 않은 환경(demo 모드)에서도 빌드 에러 없이 동작하도록 `await import('@portone/browser-sdk/v2')` 사용. 실제로는 npm 설치가 완료되어 있으므로 정상 동작한다.

## 6. 알려진 제한 사항

1. **상품 이미지 없음** — 현재 SVG 아이콘으로 대체. 실제 상품 이미지가 준비되면 `thumbnail_url` 필드 활용 가능.
2. **회원 시스템 없음** — 비회원 주문만 지원. Supabase Auth 연동은 4차 개발에서 추가 예정.
3. **주문 내역 조회 없음** — OrderConfirmation은 결제 직후 한 번만 표시. 마이페이지/주문내역 페이지는 미구현.
4. **환불/취소 기능 없음** — 관리자 대시보드에서 수동 처리 필요.
5. **상품 가격이 임시** — 실제 가격으로 교체 필요.
