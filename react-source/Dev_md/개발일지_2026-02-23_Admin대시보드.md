## 1. 작업 개요

기존 관리자 기능(블로그/갤러리/강의계획서/상품 CRUD)이 각 페이지에 분산되어 있어, `/admin` 경로 하위에 사이드바 레이아웃의 **통합 관리자 대시보드**를 신규 구축했다. 모든 콘텐츠와 상거래 관리 기능을 하나의 인터페이스에서 제공한다.

## 2. 기술 결정 사항

| 항목 | 선택 | 사유 |
|------|------|------|
| 레이아웃 분리 | Route 레벨에서 `/admin/*` vs `/*` 분기 | Navbar/Footer 없이 독립 사이드바 레이아웃 적용 |
| 기존 CRUD 재사용 | boardStorage / productStorage / supabase.js | 새 API 불필요, 기존 함수 100% 재활용 |
| 공통 테이블 | AdminDataTable 컴포넌트 | 검색/정렬/페이지네이션 내장으로 8개 페이지에서 재사용 |
| 글쓰기/수정 | 기존 페이지(`/community/blog/write` 등)로 이동 | 중복 구현 방지 |
| 권한 체크 | 기존 AdminGuard 컴포넌트 | 이메일 기반 관리자 체크 로직 그대로 사용 |
| 사이드바 접기 | CSS width 전환 + 로컬 상태 | 복잡한 라이브러리 없이 순수 CSS 구현 |
| 모바일 대응 | 오버레이 + 슬라이드인 | 768px 이하에서 사이드바 숨김 후 토글 |

## 3. 구현 상세

### 3.1 App.jsx 구조 변경 (이중 레이아웃)

**변경 전:**
```
App → Navbar + main(모든 Routes) + Footer
```

**변경 후:**
```
App → Routes
  ├── /admin/* → AdminGuard → AdminLayout (사이드바+헤더+서브라우트)
  └── /*       → PublicLayout (기존 Navbar+Routes+Footer 추출)
```

기존 Navbar + Footer 레이아웃을 `PublicLayout.jsx`로 분리 추출하고, App.jsx는 최상위 라우트 분기만 담당하도록 변경했다.

### 3.3 AdminDataTable 공통 컴포넌트

모든 관리 페이지에서 재사용하는 데이터 테이블. Props 기반으로 동작:

- `columns` — `{ key, label, width?, className?, render? }[]`
- `data` — 데이터 배열
- `loading` — 로딩 상태
- `searchKeys` — 검색 대상 필드명 배열
- `actions` — `(row) => JSX` 행별 액션 버튼
- `pageSize` — 페이지당 항목 수 (기본 20)
- `expandRow` — `(row) => JSX` 행 확장 (주문 상세 등)

내장 기능: 실시간 검색 필터, 컬럼 클릭 정렬 (문자열/숫자), 페이지네이션.

### 3.4 adminStorage.js

대시보드 통계 전용 유틸리티:

```js
getDashboardCounts()  // 7개 테이블 동시 count (Promise.allSettled)
getAllUsers()          // user_profiles 전체 조회
getAllOrders()         // orders + order_items 조인 조회
getRecentOrders(n)    // 최근 주문 N건
```

`Promise.allSettled`를 사용해 일부 테이블 접근 실패 시에도 나머지 카운트는 정상 표시되도록 구현.

### 3.5 각 관리 페이지 기능

| 페이지 | 기능 |
|--------|------|
| AdminDashboard | 7개 통계 카드 + 최근 주문 5건 + 최근 콘텐츠 5건 |

| AdminProducts | 상품 목록 + 품절 토글 + 활성 토글 + 수정 |
