## 세션 1: Supabase 스키마/설정 점검 및 수동 처리 항목 도출

### 작업 배경

프론트엔드 코드와 Supabase DB 스키마 간 정합성을 점검하고, SQL 스크립트 또는 수동 처리가 필요한 항목을 도출.

### 점검 대상 파일

| 파일 | 역할 |
|------|------|
| `supabase_schema.sql` | 이커머스 DB 스키마 (products, orders, order_items) |

| `utils/supabase.js` | 프론트엔드 주문 CRUD |
| `utils/auth.js` | 인증 헬퍼 (OAuth, signup, profile) |
| `contexts/AuthContext.jsx` | 인증 상태 관리 + check_user_status RPC |

### 발견된 SQL 이슈 (4건)

| # | 항목 | 상세 |
|---|------|------|
| 1 | `orders` 테이블 컬럼 누락 | `cancelled_at TIMESTAMPTZ`, `cancel_reason TEXT` — 코드에서 사용하나 스키마에 미정의 |
| 2 | `orders.payment_status` CHECK 불일치 | 코드: `'refunded'` 사용 / 스키마: `('pending','paid','failed','cancelled')` |
| 3 | `orders` UPDATE RLS 정책 미존재 | 프론트엔드 주문 상태 변경 시 권한 에러 가능 |
| 4 | `user_profiles.role` CHECK 불일치 | 기본값 `'member'` / CHECK: `('user','admin')` |

### SQL 스크립트

```sql
-- 1. orders 누락 컬럼
ALTER TABLE public.orders
  ADD COLUMN IF NOT EXISTS cancelled_at TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS cancel_reason TEXT;

-- 2. payment_status CHECK 보완
ALTER TABLE public.orders DROP CONSTRAINT IF EXISTS orders_payment_status_check;
ALTER TABLE public.orders ADD CONSTRAINT orders_payment_status_check
  CHECK (payment_status IN ('pending', 'paid', 'failed', 'cancelled', 'refunded'));

-- 3. orders UPDATE 정책
CREATE POLICY "orders_update" ON public.orders
  FOR UPDATE USING (
    user_id = auth.uid()
    OR EXISTS (SELECT 1 FROM public.user_profiles WHERE id = auth.uid() AND role = 'admin')
  );

-- 4. user_profiles role CHECK 보완
ALTER TABLE public.user_profiles DROP CONSTRAINT IF EXISTS user_profiles_role_check;
ALTER TABLE public.user_profiles ADD CONSTRAINT user_profiles_role_check
  CHECK (role IN ('member', 'admin'));
```
