## 세션 1: 로그인 장애 복구 + Supabase fallback 하드코딩

### 작업 배경
seoultech 등 여러 학습 사이트에서 로그인 시 **"Supabase not configured"** 발생.
원인: Vite 빌드 시점에 `import.meta.env.VITE_SUPABASE_*`가 주입돼야 하는데, fallback이 없어
env 없는 빌드에서 client가 `null`이 됨. (전 사이트가 단일 Supabase 프로젝트 `hcmgdztsgjvzcyxyayaj` 공유)

### 조치
`src/utils/supabase.ts`에 공용 프로젝트 URL/anon key **fallback 하드코딩** — env 미주입 빌드에서도 로그인 보장.
anon key는 브라우저 노출용 공개 키이며 RLS로 보호되므로 안전.

```ts
const supabaseUrl =
  import.meta.env.VITE_SUPABASE_URL || 'https://hcmgdztsgjvzcyxyayaj.supabase.co';
const supabaseAnonKey =
  import.meta.env.VITE_SUPABASE_ANON_KEY || 'eyJhbGci...'; // 공용 anon key
```

> ※ 동일 패턴으로 형제 사이트 11개(ai-free·data·koreatech·kdn2026·kdn24·kdn02/03/20/22/28/29)도 보강.

## 세션 4: 최상위 관제 — 사이트 종합 현황 + 단일 레지스트리(SSOT)

### ① 단일 소스 `constants/siteRegistry.ts` 신설
전체 **120개 하위 사이트의 단일 소스(SSOT)**. `{ name, domain, url, color, hasPayment, ordersTable, ordersSite }`.
`SITE_OPTIONS`·`SITE_COLOR_MAP`이 여기서 파생 → **신규 사이트는 레지스트리 1곳만 수정**하면 자동 반영.

### ② 사이트 현황 → 종합 현황 (`AdminSites.tsx` 전면 개편)
- 사이트별 **회원 · 최근 7일 · 방문(90일) · 주문 · 매출** 통합 테이블 + 요약 카드 6종
- **라이브 상태 점검**: no-cors 핑(배치 8개씩)으로 120개 사이트 접속 점검 → 정상/응답없음 표시 (장애 조기 발견)
- 행별 **빠른 관리 링크**(사이트 열기↗ · 회원 보기) + **CSV 내보내기** + 정렬(회원/방문/매출/신규/이름)
- 매출 집계: `getAllOrdersAll()`이 12개 결제 사이트 주문테이블(orders·eh_orders·ah_orders 등)을 `site` 태그로 통합

## 📌 배포 절차 (중요)

www Pages는 **gh-pages 브랜치**에서 서빙(main root 자산은 stale·미사용).
gh-pages `assets/images/`에 프로덕션 이미지가 별도 존재 → **naive `gh-pages -d dist`는 이미지를 삭제함**.

올바른 배포:
1. `react-source`에서 `.env`(Supabase + PortOne 4개 변수) 갖춘 채 `npm run build`
2. gh-pages 브랜치를 받아 `assets/` 최상위 **js/css만 교체**, `assets/images`는 **보존**
3. 루트 파일(index.html·404.html·favicon·robots·sitemap)·CNAME(www.dreamitbiz.com) 갱신 후 push
